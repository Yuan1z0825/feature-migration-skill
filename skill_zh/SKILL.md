# 功能迁移工作流 (Feature Migration Workflow)

将开源项目的功能精准、严谨地迁移到本地项目的四阶段工作流。

## 输入要求

- **GitHub 链接**（必需）：目标开源项目的 GitHub URL
- **目标功能**（必需）：要迁移的具体功能描述

## 工作流总览

```
阶段一: 本地上下文对齐 → 输出集成环境特征 → 等待确认
阶段二: 开源全貌调研   → 输出技术全貌图谱 → 等待确认
阶段三: 可行性分析     → 输出分析报告     → 等待确认
阶段四: 源码确认与落地 → 生成方案+修改代码
```

---

## 阶段一：本地上下文对齐 (Context Alignment)

### 目标
深度理解本地项目的技术环境，识别集成约束。

### 执行步骤

1. **目录结构分析**
   - 使用 `Glob` 扫描项目目录结构
   - 识别项目根目录的关键文件夹和文件

2. **配置文件解析**
   - 读取依赖配置文件：
     - Python: `requirements.txt`, `pyproject.toml`, `setup.py`, `setup.cfg`
     - Node.js: `package.json`, `yarn.lock`, `pnpm-lock.yaml`
     - Rust: `Cargo.toml`, `Cargo.lock`
     - Go: `go.mod`, `go.sum`
   - 提取依赖版本信息

3. **核心入口文件分析**
   - 读取主入口文件：`main.py`, `app.py`, `__main__.py`, `index.js`, `src/main.rs` 等
   - 识别框架初始化方式

4. **框架使用模式搜索**
   - 使用 `Grep` 搜索关键框架导入模式
   - 识别 API 风格、状态管理方式、配置管理模式

5. **代码风格识别**
   - 抽样读取 2-3 个代表性源文件
   - 识别命名规范、注释风格、错误处理模式

### 输出格式

```markdown
## 集成环境特征

### 项目类型
[Web应用/CLI工具/库/数据分析项目/...]

### 核心框架
- 框架名称: [Flask/Django/FastAPI/React/Vue/...]
- 版本: [x.x.x]

### 目录结构
[简要描述目录组织方式]

### 关键模块
- [模块1]: [职责]
- [模块2]: [职责]
- ...

### 技术栈约束
| 约束类型 | 详情 | 风险等级 |
|---------|------|---------|
| Python版本 | 3.8+ | 低 |
| 依赖冲突 | xxx与yyy版本冲突 | 高 |
| ... | ... | ... |

### 代码风格
- 命名规范: [snake_case/camelCase/...]
- 类型注解: [有/无]
- 文档风格: [Google/NumPy/Sphinx/...]
```

### 交互

输出后询问用户：
> **阶段一完成。请确认本地环境分析是否准确？是否需要补充或修正？确认后进入阶段二。**

---

## 阶段二：开源全貌调研 (Macro Exploration)

### 目标
纵览开源项目的核心架构，识别可复用组件。

### 执行步骤

1. **DeepWiki 调研**
   - 从 GitHub URL 提取 owner/repo
   - 调用 `mcp__deepwiki__get-deepwiki-index` 获取项目索引
   - 调用 `mcp__deepwiki__get-deepwiki-page` 读取核心页面：
     - 架构文档
     - API 文档
     - 核心模块文档

2. **GitHub 仓库分析**
   - 使用 `gh repo view` 获取仓库基本信息
   - 使用 `gh api` 获取目录结构
   - 读取 README.md 了解项目定位

3. **核心模块定位**
   - 识别与目标功能相关的核心模块
   - 分析模块间的依赖关系
   - 标注关键接口和数据结构

### 输出格式

```markdown
## 技术全貌图谱

### 项目定位
[一句话描述项目核心价值]

### 整体架构
[架构图或文字描述]

### 核心模块
| 模块名 | 职责 | 与目标功能关联度 |
|--------|------|-----------------|
| [模块1] | [...] | 高/中/低 |
| ... | ... | ... |

### API 接口设计
[关键 API 签名和调用方式]

### 数据流
[核心数据如何流转]

### 可复用组件
[可直接复用/需适配/不可复用的组件列表]

### 与本地项目交互点
- 接口1: [描述]
- 接口2: [描述]
- ...
```

### 交互

输出后询问用户：
> **阶段二完成。开源项目全貌已分析完毕，请确认是否继续进行可行性分析？**

---

## 阶段三：集成可行性分析 (Feasibility Study)

### 目标
评估集成的技术可行性，识别风险，设计集成方案。

### 执行步骤

1. **MVP 代码集识别**
   - 基于目标功能，确定最小必需代码
   - 列出必须迁移的文件和依赖

2. **兼容性风险评估**
   - 对比本地技术栈与开源项目技术栈
   - 识别版本冲突、API 不兼容、范式差异
   - 评估依赖引入风险

3. **集成逻辑设计**
   - 设计模块间的调用关系
   - 定义适配层（如需要）
   - 规划配置管理方式

### 输出格式

```markdown
## 可行性分析报告

### 实现路径
[整体迁移策略描述]

### MVP 代码集
| 文件/模块 | 用途 | 是否必需 |
|----------|------|---------|
| [...] | [...] | 是/否 |
| ... | ... | ... |

### 兼容性风险
| 风险项 | 描述 | 影响程度 | 缓解方案 |
|--------|------|---------|---------|
| [风险1] | [...] | 高/中/低 | [...] |
| ... | ... | ... | ... |

### 依赖分析
- 新增依赖: [列表]
- 版本冲突: [列表及解决方案]

### 集成设计
#### 模块结构
[迁移后的模块结构设计]

#### 接口适配
[需要适配的接口及适配方式]

#### 配置管理
[配置项的设计和管理方式]

### 工作量估算
- 文件新增: [N] 个
- 文件修改: [N] 个
- 预计工作量: [小/中/大]
```

### 交互

输出后询问用户：
> **阶段三完成。可行性分析已输出，请确认方案是否可行？确认后将深入源码并生成实现。**

---

## 阶段四：源码精准确认与落地 (Micro Verification & Implementation)

### 目标
深入验证源码细节，生成符合本地规范的实现代码。

### 执行步骤

1. **源码深度分析**
   - 使用 `gh api` 获取关键文件内容
   - 交叉验证 DeepWiki 描述与实际代码
   - 确认：
     - 变量定义和类型
     - 异常处理逻辑
     - 上下文依赖（Context/Hooks/Utils）
     - 边界条件和边界处理

2. **实现方案生成**
   - 基于本地代码规范编写代码
   - 适配本地项目的命名、风格、模式
   - 添加必要的注释和文档

3. **代码落地**
   - 使用 `Write` 创建新文件
   - 使用 `Edit` 修改现有文件
   - 更新配置文件（依赖、配置项等）

4. **验证检查**
   - 检查语法正确性
   - 确认导入路径正确
   - 验证接口签名一致

### 输出格式

```markdown
## 实现方案

### 核心源码验证
[关键源码片段及分析]

### 代码实现

#### 新增文件
- `[文件路径]`: [用途说明]

#### 修改文件
- `[文件路径]`: [修改内容说明]

### 关键代码片段
[展示核心实现代码]

### 使用说明
[如何在本地项目中使用新功能]

### 后续工作
- [ ] 单元测试
- [ ] 集成测试
- [ ] 文档更新
- [ ] ...
```

---

## 注意事项

1. **渐进式确认**：每个阶段输出后必须等待用户确认，不可跳过
2. **代码规范对齐**：生成的代码必须符合本地项目的代码风格
3. **最小化修改**：优先复用本地现有模式，避免引入不必要的复杂性
4. **依赖审慎**：新增依赖需明确必要性，避免依赖膨胀
5. **文档同步**：修改代码时同步更新相关注释和文档

## 错误处理

- 若 DeepWiki 无该项目索引，跳过阶段二的 DeepWiki 部分，仅使用 GitHub 分析
- 若 GitHub 访问受限，提示用户提供源码或文档
- 若目标功能与开源项目不匹配，在阶段二或阶段三明确指出并建议调整
